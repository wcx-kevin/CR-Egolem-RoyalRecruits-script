# E-Golem 规则研究报告

## 结论

- 我把仓库里的 e-golem 牌组解释为运行时自动化版本，而不是泛指人类常见的同名流派。具体以 `src/cr_bot/config/decks.py:27` 的 `elixir_golem` 配置和 `src/cr_bot/gameplay/format2_elixir_golem.py:12` 的脚本为准。
- 这套牌的实际构成是：`Elixir Golem`、`Night Witch`、`Goblin Machine`、`Skeleton King`、`Rage`、`Arrows`、`Skeletons`、`Archers`。`decks.py` 里还允许少量模板别名，但自动化落子脚本只按这 8 张牌执行。
- 这份报告优先面向自动化：脚本目标不是“最强人类打法”，而是“稳定、可循环、可恢复”的固定流程。

## 牌组身份

- 牌组核心身份是“重型前排 + 持续支援 + 低费重置”。`Elixir Golem` 负责开局主压，`Night Witch` 和 `Goblin Machine` 负责把压力转成持续输出，`Skeleton King` 提供中后期收尾与反打承接。
- `Rage` 的定位不是独立进攻牌，而是把重单位和支援单位的输出窗口压缩到更短时间里。
- `Skeletons`、`Archers`、`Arrows` 是循环与补洞组件：它们的主要任务不是赢线，而是让主牌尽快重新可用，同时避免手牌堆死。

## 实际攻击分期

- 单倍圣水期：偏保守，只打一套标准开局链。`deck_runtime.py:29` 里先跑 1x 序列，再判断时间是否仍早于 90 秒，若是才补第二套 1x。
- 双倍圣水期：进入主进攻窗口，开始稳定加入 `Skeleton King`，并通过更密集的支援和更快的重压节奏提高过塔概率。
- 三倍圣水期：进入爆发堆叠，允许更高频率的 `Elixir Golem` 重放、`Arrows` 清场和双支援连打，目标是让对手没有足够回合做完整解法。

## 卡牌职责与默认脚本用法

### `Elixir Golem`

- 角色：主坦克、主节奏锚点、主要换费触发器。
- 默认用法：1x 和 2x 里都先作为起手牌；3x 里先后出现两次，说明脚本把它当作可重复的压力源，而不是一次性终结牌。
- 位置：脚本多用 `lane_offset` 控制同侧落点，表示同一路开压；3x 阶段会改到 `4 + lane_offset` 和 `9 - lane_offset` 这类更激进的位置。具体屏幕位置由 `placeCard.py:38` 的固定坐标表决定。

### `Night Witch`

- 角色：主输出支援、死亡价值来源、对重坦压线后的续航核心。
- 默认用法：紧跟 `Elixir Golem` 落下，始终和主坦同侧成链。
- 位置：1x/2x/3x 都是同侧跟进；这表示脚本把她视为“必须绑定主坦”的单位，而不是独立桥头消耗牌。

### `Goblin Machine`

- 角色：远程压制、补伤害、对空对地补位。
- 默认用法：通常放在对侧或偏后位置，制造横向压力，而不是和主坦完全堆在一条线上。
- 位置：1x 里偏 `9 - lane_offset`，2x 里偏 `8 + lane_offset`，3x 里仍是 `8 + lane_offset`。这说明脚本更重视它的安全输出位，而不是近身换血。

### `Skeleton King`

- 角色：中后期英雄坦、残局收口、对换反打的核心。
- 默认用法：只在双倍和三倍阶段出现，说明脚本默认不拿他做前期节奏牌，而是把他留给更高密度的交换窗口。
- 位置：2x 里在 `1 - lane_offset`，3x 里在 `3 - lane_offset`。这表示他通常被放在偏防守或偏中路的安全位，再转入进攻。

### `Rage`

- 角色：节奏放大器、终结器、进攻窗口压缩工具。
- 默认用法：不是随便补费，而是绑定在主波次附近使用。
- 位置：1x 和 2x 都在 `10 + lane_offset`，3x 也保留在这个区间，说明它主要服务于同路推进后的临门一脚。

### `Skeletons`

- 角色：最低成本循环、临时拉扯、补过牌。
- 默认用法：优先拿来补手、压节奏、把主卡重新洗回去。
- 位置：1x 里在 `4 + lane_offset`，2x 里在 `5 - lane_offset`，3x 里在 `5 - lane_offset` 和后面的 `4 + lane_offset`。这类落点说明它们既承担 cycle，也承担小范围拉扯。

### `Archers`

- 角色：低费空地补位、稳定输出、对过载的保险。
- 默认用法：在 1x 末尾和 2x 中后段作为补刀和防守衔接，3x 里继续作为固定补位单位。
- 位置：1x 里是 `4 + lane_offset`，2x 里是 `7 - lane_offset`，3x 里是 `7 - lane_offset`。这意味着脚本把她们当成“离主坦稍远、但还能覆盖战线”的安全输出卡。

### `Arrows`

- 角色：低成本清场、打断敌方轻量防守、补 cycle。
- 默认用法：只在更高节奏阶段明显出现，说明它更像“节奏清算器”而不是前期常驻牌。
- 位置：3x 阶段出现于 `11 - lane_offset`，即更靠后的清场位。

## 站位与落点

- 脚本不直接写“桥头、后场、同路深位”这种语义，而是用 `placeCard.py:40-41` 的 12 个固定落点索引来落子。
- 因此，这里的站位解释应视为 `Heuristic`：它描述的是“脚本想达到的战术意图”，不是地图学上的精确坐标语义。
- 对自动化来说，最重要的是保持“同一路主压 + 旁路支援 + 低费补洞”的相对关系，而不是记死某个点位的手感。

## 英雄处理：Skeleton King

- 代码里没有任何技能点击、英雄大招触发或 champion ability 自动释放逻辑。也就是说，仓库当前只会把 `Skeleton King` 当作一张会下场的牌，不会主动点能力。
- 所以这里的英雄处理只能做近似：`Estimate` 为“把他当成中后期可承压的第二前排”，`Heuristic` 为“等他已经吃到足够亡灵/魂值、或对手单位密度足以换来明显收益时，再考虑外部触发能力”。
- 如果没有外部技能控制能力，就应把他的价值理解为“站场与收残”而不是“必须依赖大招才能成立的核心输出来牌”。
- 2x 阶段他出现在第 3 步，说明脚本希望他在主坦之后入场；3x 阶段他被提前到第 4 步，说明高密度对局里可以更早把他放上场，但仍然不应过早裸交。
- `Estimate`：若做自动化补完，Skeleton King 的能力窗口应更偏向“前排已吸引火力、后排已有小兵死亡、敌方反打单位进入魂收益区”这三个条件同时满足时。

## 循环逻辑

- `changeCycle.py:58` 的 e-golem 循环判定很明确：要让 `Elixir Golem` 处于可用位，同时让 `Night Witch`、`Goblin Machine`、`Skeleton King`、`Rage` 落在前 5/6/7 格附近。
- 当循环没准备好时，脚本优先用廉价牌清轮：`Rage`、`Skeletons`、`Archers`、`Arrows`，见 `changeCycle.py:73`。
- 如果廉价牌都不能推进，脚本会尝试用 `Unknown` 触发识别刷新，再不行才回到更重的补救牌。
- 这个设计的本质是“为主压波次服务的洗牌器”，不是为了在每个回合都追求最优交换。

## 防溢出 / 防过载 / 防死推

- 防溢出：`card_tracker.py:11-21` 通过识别延迟、最小置信度、重复确认和定期刷新，避免把不确定手牌当成确定手牌。自动化上，这比“立刻下最大牌”更重要。
- 防过载：`comCycle.py:80` 的 `use_card` 会把打出的牌挪到不可用队列尾部，再从队头补新牌，保证循环结构不被破坏。只要这个队列逻辑稳定，脚本就不会因为盲目重复而把自己卡死。
- 防过冲：遇到不完整支援时，不要为了继续洗牌而硬塞 `Elixir Golem`。没有 `Night Witch` / `Goblin Machine` 的支援链时，主坦容易变成死推。
- 防死推：当敌方压力不足、或者我方手牌还没重整完成时，优先打低费过渡牌，而不是直接把重牌推进去。脚本层的正确思路是“先让压力形成闭环，再让坦克进场”。
- `card_tracker.py:231` 的 `reconcile_after_play` 也体现了这个方向：出牌后先更新循环，再做可见区复核，防止状态漂移。

## 分阶段脚本计划

### 单倍圣水

- 目标：稳住节奏，拿信息，建立第一次主波次。
- 默认流程：先 `Elixir Golem`，再 `Night Witch`，然后 `Goblin Machine`，接着 `Rage`，最后用 `Skeletons` 和 `Archers` 过牌。
- 自动化原则：不要在单倍里把所有高费牌一次性摊平；保留后续回合再打一波的空间。

### 双倍圣水

- 目标：进入主压窗口，开始稳定带 `Skeleton King`。
- 默认流程：`Elixir Golem` -> `Night Witch` -> `Skeleton King` -> `Goblin Machine` -> `Skeletons` -> `Archers` -> `Rage`。
- 自动化原则：此时可以更主动，但仍要保留一张低费牌做 cycle 保险，否则容易在下一轮断压。

### 三倍圣水

- 目标：最大化密度，压迫对手的全部解牌窗口。
- 默认流程：`Elixir Golem` -> `Night Witch` -> `Goblin Machine` -> `Skeleton King` -> `Skeletons` -> 再次 `Elixir Golem` -> `Archers` -> `Rage` -> `Arrows` -> `Skeletons`。
- 自动化原则：这里最适合“连续波次”，但也最容易把自己推成空手。应把 `Arrows` 和廉价随从当作重新启动引擎，而不是单纯的补伤害。

## 自动化执行建议

- 把这套牌理解成“固定开局 + 条件重启”的脚本，不要按人类天梯思路去追求每一手的局部最优。
- 主坦只有在支援链和循环条件同时满足时才值得落下；这是这套脚本里最重要的胜率来源。
- 所有不确定时序都应标记为 `Estimate` 或 `Heuristic`，尤其是 `Skeleton King` 的能力时机和桥头压制节奏。
- 如果后续要细化，最值得补的不是更多进攻花样，而是更稳定的英雄技能窗口和更严格的防死推门槛。
